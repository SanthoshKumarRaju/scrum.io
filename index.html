10‑Minute Quick‑Start (Scrum in Jira)

  1. Create a Company‑managed Scrum project in Jira → enable backlog & sprints.
  2. Capture work as User Stories (value‑focused) and Tasks (technical work). Link to an Epic.
  3. Estimate stories in story points. Define team capacity for the sprint.
  4. In Sprint Planning: pull top‑priority stories into the Sprint Backlog; break into sub‑tasks.
  5. Work on the board (To Do → In Progress → In Review → Done). Respect WIP limits.
  6. Use Daily Scrum to unblock; update board, remaining time, and links.
  7. Track with Burndown (sprint), Cumulative Flow (flow), and Velocity (predictability).
  8. End sprint with Review (demo the increment) and Retro (one improvement per sprint).

Scrum Basics

Scrum is an empirical, lightweight framework for complex work. It relies on transparency, inspection, and adaptation. Short iterations called sprints (1–4 weeks) deliver a usable increment and create consistent feedback loops.

Pillars

  • Transparency
  • Inspection
  • Adaptation

Values

  • Commitment
  • Focus
  • Openness
  • Respect
  • Courage

Timebox

  • Sprint: 1–4 weeks
  • Events are timeboxed
  • Definition of Done governs quality

Roles

Product Owner

Maximizes value, owns backlog ordering, clarifies outcomes, accepts work.

Scrum Master

Coaches Scrum, removes impediments, fosters flow, facilitates events.

Developers

Cross‑functional team building the increment; collectively own quality and estimates.

Artifacts

  • Product Backlog → ordered list of everything to build. Ownership: Product Owner.
  • Sprint Backlog → selected items + plan to deliver them.
  • Increment → integrated, potentially shippable output that meets the Definition of Done.
Definition of Done: shared quality checklist that every item must satisfy to be considered done (tests, docs, security, deployment, etc.).

Jira Setup (Step‑by‑Step)

  1. Create Project: Company‑managed Scrum. Name it and connect to your issue scheme.
  2. Issue Types: Epic, Story, Task, Bug, Sub‑task. Keep it lean to start.
  3. Workflow: To Do → In Progress → In Review → Done. Add Blocked if helpful; avoid over‑engineering.
  4. Fields: Story Points (number), Components, Labels, Fix Version, Assignee. Hide non‑essential fields for clarity.
  5. Board: Enable backlog; set swimlanes by Epic; quick filters for assignee = currentUser(), type != Sub-task, and priority in (High, Highest).
  6. Permissions: Keep editing restricted; anyone can comment; define roles (Admins, Members, Viewers).

Sprint Planning Flow

  1. Refine backlog (ongoing): ensure items have clear acceptance criteria and estimates.
  2. Set goal: a single Sprint Goal describing the outcome/value.
  3. Forecast: use past velocity (avg. points) and current capacity to select work.
  4. Break down: decompose stories into sub‑tasks (design, FE, BE, tests, docs, ops).
  5. Plan how: identify dependencies, risks, and spike unknowns.
  6. Start sprint: timebox, assign initial owners, confirm DoD.

Distributing Work Across the Team

Capacity‑based Forecasting

  • Compute each member’s available hours (days × hours − meetings − PTO).
  • Map story points to capacity using historical throughput.
  • Prefer pull (devs take next highest‑value item) over push assignment.

Work Allocation Rules

  • Respect WIP limits per column/swimlane.
  • Split large stories (> 8 points) before sprint.
  • Pairing/mobbing for risky or cross‑skill learning.

Useful Jira Structures

  • Epics to group value slices; Versions for releases.
  • Components to hint ownership (e.g., API, Mobile, DevOps).
  • Labels for cross‑cutting themes (perf, security).
  • Sub‑tasks to spread work without bloating the board.

Quick Filters (JQL)

assignee = currentUser() AND sprint in openSprints() ORDER BY updated DESC
labels = perf AND statusCategory != Done
component in ("API","Mobile") AND type != Sub-task
blocked is not EMPTY OR "Blocked" in labels

Executing the Sprint

  • Start each day by updating the board and remaining time/points.
  • Use Daily Scrum to inspect flow: what moved to Done? what’s blocked?
  • Swarm on blocked/high‑risk items first; respect WIP limits.
  • Keep PRs small; automate checks; link commits and PRs to the Jira issue key.
Tip: Add a Blocked status with a red card color and a board quick filter status = Blocked to trigger fast swarming.

Review & Retro

Review

  • Demo working software against acceptance criteria.
  • Invite stakeholders; capture feedback as new backlog items.
  • Update release notes and increment documentation.

Retro

  • Inspect people, process, tools.
  • Pick one improvement with an owner & due date.
  • Track retro actions in Jira (issue type: Task, label: retro).

Metrics & Reports (What to Watch)

  • Lead/Cycle Time: how long issues take. Aim to reduce variability.
  • Work Item Age: how long current items have been open; great for daily inspection.
  • Escaped Defects: quantifies quality; pair with DoD checks.

Definition of Ready (DoR)

  • Clear value & outcome
  • Acceptance criteria written
  • Dependencies identified
  • Estimated (points)
  • Small enough (≤ 1 sprint)

Definition of Done (DoD)

  • Code reviewed & merged
  • Automated tests pass
  • Security & performance checks
  • Docs & release notes updated
  • Deployed to target env

Common Anti‑Patterns

  • Over‑customized workflows that slow the team
  • Assigning all work during planning (kills pull and flow)
  • Carrying over large stories every sprint
  • Using story points as individual performance metrics
  • “Mini waterfalls” within a sprint

FAQ

How long should our sprint be?

Start with two weeks. If feedback cycles are too slow, try one week; if overhead is too high, consider three.

Do we need story points?

No, but they help forecast and communicate uncertainty. Flow metrics can work too; be consistent.

How do we manage bugs?

Track as their own issue type; cap a weekly bug budget (e.g., 20% of capacity) or prioritize criticals immediately.
